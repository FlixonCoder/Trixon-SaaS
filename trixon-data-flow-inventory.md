# Trixon — Data Flow Inventory (Current State)
### Reference document for privacy policy drafting — not legal advice

This is a factual inventory of what data Trixon currently collects, where it's stored, and which third parties it flows to, based on the actual implemented architecture as of this point in development. It should be reviewed by a lawyer before being relied on for legal compliance, and updated whenever new data collection is added (including the analytics feature discussed alongside this document).

---

## 1. Account & Identity Data

| Data | Source | Stored In | Purpose |
|---|---|---|---|
| Email, password (or GitHub OAuth identity) | Supabase Auth at signup | Supabase `auth.users` | Authentication |
| Full name, company name, role, primary goal | Onboarding form | `profiles` table | Personalization, product use |
| Plan tier (free/builder/founder) | System-assigned | `profiles.plan` | Feature gating (currently bypassed — `BETA_MODE=true`) |

---

## 2. Third-Party Account Connections (Sensitive)

| Data | Source | Stored In | Purpose |
|---|---|---|---|
| GitHub/GitLab OAuth access token | OAuth flow | `vcs_connections.access_token` (encrypted at rest, AES-256 per spec — **verify this is actually implemented**, flagged for security review) | Authenticating to fetch repo data |
| Platform user ID, username | OAuth flow | `vcs_connections` | Identifying the connected account |

---

## 3. Repository & Codebase Data — the most sensitive category

| Data | Source | Stored In | Purpose |
|---|---|---|---|
| Repo metadata (name, URL, platform, branch) | GitHub/GitLab API | `projects` table | Identifying which repo is being tracked |
| **Actual source code content** (selected files: README, entry points, route files, capped at 30 files/8000 chars each) | Fetched live from GitHub/GitLab API during analysis | Transiently used for LLM context; **persisted** in `code_snapshots` table (added for chat code-search feature) | Generating reports; enabling chat to answer code-specific questions |
| Static analysis results (languages, frameworks, dependency names, API route paths, env var **names** — never values, detected third-party services, DB model names, git commit stats) | Computed locally from fetched repo content | `analyses` table (various columns) | Powering reports and dashboard |
| **Repo code sent to Groq API** | Sent as part of LLM prompts during report generation and chat | Not stored by Trixon beyond the above — but transmitted to Groq's servers as a third-party processor | Core product function (AI analysis) |

**This is the single most important disclosure point for the privacy policy** — users' proprietary source code is transmitted to a third-party AI provider (Groq) as part of normal product operation.

---

## 4. Generated Analysis Output

| Data | Source | Stored In | Purpose |
|---|---|---|---|
| Reports (executive summary, architecture, tech debt, security, scalability, onboarding, investor) | Generated by Groq LLM calls | `reports` table | Core product deliverable |
| Health/category scores | Derived from report content | `analyses` table | Dashboard display |
| Action items (titles, descriptions, severities, file paths, AI-generated fix prompts) | Extracted from reports | `action_items` table | Core product deliverable |
| Snapshot diffs and AI-generated changelog summaries | Computed by diff engine + 1 LLM call | `analysis_diffs` table | Continuous-tracking feature |
| Commit metadata (SHA, message, **author**) when webhook-triggered | GitHub/GitLab push webhook payload | `analyses` table | Tying snapshots to commits |

**Note on commit author data:** if multiple people commit to a connected repo, Trixon processes the *commit author's* name/email even though they may not be the Trixon account holder and haven't directly consented to Trixon processing their data. Worth flagging to your lawyer — this is a common SaaS pattern (similar to any CI/CD tool), but should be acknowledged rather than overlooked.

## 6. Webhook & Integration Data

|---|---|---|---|
| Webhook ID, webhook secret | Created via GitHub/GitLab API | `webhook_connections` table | Verifying authentic push events |
| Incoming push payloads (processed transiently, not stored verbatim) | GitHub/GitLab | Not persisted as raw payloads — only extracted fields (SHA, message, author) are stored | Triggering automatic analysis |

## 7. Third-Party Data Processors (Where Data Leaves Trixon's Direct Control)

| Processor | What It Receives | Purpose |
|---|---|---|
| **Groq** | Source code excerpts, report context, chat messages | AI analysis and chat generation — **the most significant third-party data flow** |
| **Supabase** | Nearly all account, project, and analysis data | Database, authentication, file storage |
| **GitHub / GitLab** | OAuth tokens (sent back to them for API calls); receives nothing new from Trixon beyond standard API requests | Source of repo data |
| **Vercel** | Frontend traffic, request metadata | Frontend hosting |
| **Render** | Backend traffic, all environment secrets, request metadata | Backend hosting |
| **Stripe** (not yet active) | Would receive payment/billing data once subscription tiers launch | Future billing |
| **Resend** (if implemented) | Email addresses, notification content | Email notifications |

## 8. What Trixon Does NOT Currently Collect

- No advertising or tracking pixels
- No payment data yet (BETA_MODE bypasses billing entirely)
- No product usage analytics/telemetry (no event tracking system exists yet — see the companion prompt for the proposed addition)
- No data sold or shared with advertisers

---

## 9. Compliance Gaps to Address Before Finalizing the Privacy Policy

These aren't data flows — they're functional gaps that affect what the policy can honestly promise:

- **No account/data deletion mechanism exists.** If the privacy policy will state a right to deletion (likely required under GDPR and DPDP), there needs to be an actual way to fulfill that request — currently there isn't one built.
- **No explicit consent capture at signup.** There's no checkbox or recorded acceptance of a privacy policy/terms of service during onboarding. The footer "Privacy Policy" link exists on the marketing page but likely points to a placeholder.
- **No data export mechanism** (a "download my data" feature), which is often paired with deletion rights under the same regulations.
